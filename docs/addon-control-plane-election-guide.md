# Addon 控制面选主收口指南

本文面向 Addon 开发与测试工程师，聚焦一个通用主题：当实例角色判断、对外发布、fencing、failover / switchover 分散在多个脚本或组件里时，如何把选主职责收口到单一 control-plane。

正文只写通用方法论，不绑定某一个引擎。具体引擎实现、命令、案例结论应放在独立案例材料中。

## 先用白话理解这篇文档

这篇文档的核心，其实是在回答一个很现实的问题：

- **到底谁有最终拍板权**

如果启动脚本、watcher、probe、agent、controller 都在各自判断“现在谁是 primary”，现场就会像一辆车上有好几个人同时抢方向盘：

- 有人说该往左
- 有人说该往右
- 最后最危险的不是谁偶尔看错了，而是根本没有唯一驾驶员

所以这里说的 `control-plane 收口`，不是把所有脚本都删掉，而是把它们都降级成“报信号的人”，只保留一个真正拍板的人。

## 适用场景

这份文档适用于以下问题线：

- startup 阶段可能过早自举并暴露 primary
- `roleProbe`、watcher、bootstrap 脚本各自持有部分“谁是 primary”的判断
- endpoint / publication 切换与真实角色收敛脱节
- old primary fencing 没有单一权威执行面
- failover / switchover 的完成信号来自多个并行路径

如果一个 Addon 已经出现“多个地方都像在拍板 primary”的现象，就不该继续局部加 guard，而应先判断是否需要把职责收口到 control-plane。

## 先识别 3 类 truth 是否被拆散

排查这类问题时，先区分 3 类 truth：

1. **role truth**
- 数据面当前谁真的在承担 primary 角色

2. **publication truth**
- 系统当前把谁对外发布成 primary

3. **fencing truth**
- 旧 primary 是否已经退位、隔离，或至少不再继续对外提供冲突写路径

如果这 3 类 truth 分散在不同脚本或组件里分别判断，就容易出现：

- role 已变化，但 publication 还没收敛
- publication 已切，但 old primary 还没 fencing
- 启动脚本与 watcher 各自给出不一致结论

## 什么时候应该收口到 control-plane

出现以下任一现象时，应优先考虑 control-plane 收口，而不是继续做局部补丁：

- startup self-elect 暴露 primary 过早
- auto-switchover 在拓扑未稳定前触发
- `roleProbe` 与 watcher 的判断先后不一致
- 短时间内出现 dual-primary / role oscillation
- endpoint 切换时机不稳定
- 不同路径都能独立宣布“现在谁是 primary”

这类问题的本质通常不是某一个条件分支写错，而是**最终拍板权没有被单点定义**。

## 推荐的职责分层

### data plane

- 实例真实角色
- 复制链路状态
- 本地 readiness / liveness / probe

### signal plane

- `roleProbe`
- watcher
- 启动脚本或本地探针输出
- 其他只负责上报现象的组件

这些信号可以不稳定、可以短时冲突、可以晚到。它们只能提供输入，不能单独拍板。

### control plane

- 最终确认 primary
- 决定是否切 endpoint / publication
- 决定是否先做 fencing
- 收口 failover / switchover 完成信号

工程上最关键的一点是：**只有 control-plane 有权把输入信号变成对外最终结论。**

## 旧路径如何处理

### 退出主决策面的路径

- startup 阶段直接 self-elect 并对外暴露 primary
- 不经过统一仲裁就切 endpoint 的路径
- watcher 基于局部观察结果直接宣布新 primary 的路径

### 保留但降级为信号的路径

- `roleProbe`
- watcher
- bootstrap / startup 脚本
- 各种本地探针或快速判定脚本

这些路径可以继续存在，但只能报信号，不能继续拥有并行拍板权。

## 迁移时要先回答的问题

在进入实现前，至少先把下面的问题答清楚：

1. control-plane 依赖哪些最小输入信号确认 primary
2. 什么条件下允许发布新 primary endpoint
3. 什么条件下必须先做 old primary fencing
4. failover 与 switchover 是否共用同一套最终发布逻辑
5. 启动期短时主从抖动如何避免被误发布
6. 当 `roleProbe` 与复制拓扑短时冲突时，哪一方只是信号，哪一方是最终拍板输入

## 不要把单一能力探测面当成 promote / has-lease 的致命 gate

白话一点说，不要因为“有一个温度计没读到数”，就直接宣布“发动机坏了，整车不能开”。

很多能力其实可以从多个面看到：

- 变量
- 本地状态
- 元数据表
- 插件表

其中某一个面空了、查失败了、版本不兼容了，通常更像“这个观察窗没看清”，不一定等于“能力真的不存在”。

控制面在做 leader refresh、`has-lease`、promote 或等价判断时，经常还会顺手确认一些“能力是否已开启”的事实，例如：

- 某类复制增强能力是否已打开
- 某个插件或扩展是否已加载
- 某个运行时变量是否已经生效

这里很容易踩到一个通用坑：

- 把其中**某一个探测面**的查询失败、空结果或版本差异
- 直接当成“不能 promote / 不能持有 lease / 当前没有 leader”的致命错误

这会导致：

- 真正的 leader refresh 先被打穿
- control-plane 提前丢掉 leader
- 后续才连锁表现成 `no leader`、双边只读、对外写入口消失、甚至复制冲突

## 先区分 3 种状态

这类能力探测，至少要把结果分成 3 种，而不是简单二值化：

1. **能力已明确可用**
- 有稳定证据表明能力存在且可工作

2. **能力当前未知，但没有足够证据证明它不可用**
- 某个元数据查询没结果
- 某个可选表/插件视图为空
- 某一条探测路径受版本差异或实现差异影响

3. **能力已明确不可用，继续 promote 确实不安全**
- 有足够证据表明关键条件不满足
- 继续持有 leader 或 promote 会造成错误写入或拓扑破坏

最危险的实现方式是把第 2 类直接等同于第 3 类。

## 有多个等价探测面时，优先走最稳定、最兼容的路径

如果同一能力可以从多个面观察到，例如：

- 运行时变量
- 本地状态或 agent 已知状态
- 元数据表 / 插件表 / 系统视图

更稳妥的顺序通常是：

1. 先用最稳定、跨版本差异最小的路径
2. 再用额外探测面做补强
3. 某一个可选探测面没有结果时，先降级为“未知”，不要立即当成致命失败

也就是说：

- **缺少某一类元数据，不等于能力一定不存在**
- **单一探测面的空结果，不应直接打断 promote / has-lease**

## 当“元数据存在性”和“运行时状态”都能反映同一能力时，优先信运行时状态

白话一点说：

- 仪表盘已经亮着
- 但后台登记表没更新

这时更该先信车现在真的在跑，而不是先信后台那张表一定最权威。

所以“表里没行”很多时候只是补强证据缺失，不该直接盖过运行时已经给出的主路径证据。

实践里很常见的一种误用是：

- 先查某个插件表 / 系统视图 / 元数据表
- 如果查不到行，就直接判“能力不可用”
- 后续不再继续看运行时变量或更直接的主路径证据

这类实现对版本差异和兼容性非常脆弱，因为：

- 有些引擎会把能力暴露在变量里，但不保证元数据表一定有对应行
- 有些表本来就是补强观测面，而不是控制面契约
- “查不到”往往只能证明该查询面不稳定，不能证明能力一定关闭

更稳妥的顺序通常是：

1. **先看运行时状态**
- 当前是否已启用
- 当前是否还能继续 enable / disable
- 当前是否满足 promote / has-lease 的最小安全条件

2. **再看元数据存在性**
- 作为增强置信度的补充证据
- 或用于区分更细的实现分支

3. **元数据面无结果时先降级，不先致命失败**
- 记成 unknown / best-effort
- 继续走变量路径或其他主证据链

一句话：

- **“表里没行”更像是补强证据缺失，不该自动盖过“变量可读、状态可判、操作可继续”的主路径证据。**

## 不要把“首次启动”和“带数据重启”混成一条 prestart 路径

白话一点说：

- 空房子第一次入住
- 和住过人的房子只是重新开门

不是一件事。

如果系统把第二种误判成第一种，就会做一堆本来不该做的重动作，例如重新拉全量数据、重新走 join、重新找同步源，最后把一次普通重启拖成更大的故障。

控制面或 syncer 在 `PreStart()` / `Join()` / `SyncData()` 之前，通常都会先判断当前实例更像：

1. **首次启动 / 空盘加入**
2. **已有数据的重启实例**

这一步如果判断错了，后果往往比表面看起来更早：

- 已有数据的 secondary 被误判成“空盘”
- 代码提前进入 `SyncData()` / 全量同步 / 重建路径
- 后续又因为 leader 发现不稳定而直接 crash
- 现场最终表现成 `leader member not found`、`CrashLoopBackOff`、`no leader`

## 判断“空盘”必须看真实持久化数据路径

这类误判最常见的来源是：

- 代码拿了一个历史子目录、兼容目录或假定路径去判断“数据盘是否为空”
- 但 live 引擎真实持久化的数据根本不在那个路径上

于是就会出现：

- 实例实际上已经带着有效数据重启
- 控制面却把它当成首次 bootstrap

更稳妥的实现口径是：

- 先钉实 live 引擎真正使用的持久化数据根路径
- 空盘判断只看这个真实路径
- 不要用过时的子目录命名、实现假设或旧版本布局去推断当前是否有数据

一句话：

- **“看错目录”会把重启误判成首次启动。**

## 在进入 SyncData / 全量同步前，leader 不只是“非 nil”，还必须“有效可用”

这条可以理解成：

- 通讯录里有这个人

不等于

- 这个人的电话现在能打通，而且他还真能帮你做这件事

所以 `Leader != nil` 只说明“名单上有个对象”，不说明它现在还是有效 leader，更不说明它能安全拿来当同步源。

另一个常见坑是：

- DCS / lease 对象表面上还存在
- 代码只检查 `Leader != nil`
- 就认为当前已经有 leader，可以继续 `SyncData()`

但真实现场里，leader 可能已经：

- 过期
- 名字为空
- 无法解析到有效 member
- 或根本不能作为当前同步源使用

这类情况下，如果继续走 `SyncData()`，很容易直接撞成：

- `leader member not found`
- prestart 失败
- secondary CrashLoop

更稳妥的判断应该至少拆成两层：

1. **leader object 是否存在**
2. **leader member 是否有效且未过期**

只有第二层也成立时，才允许进入依赖 leader 的同步路径。

## 推荐的 prestart 收口方式

在 `PreStart()` / `Join()` 一类启动前逻辑里，建议固定成下面顺序：

1. 先判断当前实例是空盘首次启动，还是带数据重启
2. 这个判断只依赖真实 datadir / 持久化状态
3. 只有确实需要同步时，才进入 `SyncData()` / 全量同步路径
4. 进入同步前，再确认 leader member 有效、未过期、可解析
5. 任一条件不成立时，优先降级为等待 / 重试 / unknown，而不是直接误入错误同步路径

如果这几层不拆开，现场就会出现一种很迷惑的形状：

- primary 其实已经恢复可写
- secondary 却仍在 prestart 里因为“找不到 leader member”反复 crash
- 表面像 leader discovery 问题
- 最早破坏点其实是“带数据重启被误判成需要同步”

## invalid leader 更适合走可重试路径，不要直接把 secondary 打进 CrashLoop

当实例已经带数据重启，而当前 leader 只是不稳定、过期或暂时不可解析时，更稳妥的控制面行为通常是：

- 暂时不进入 `SyncData()` / 全量同步
- 返回等待 / 重试
- 让 secondary 在下一轮继续观察 leader 是否恢复有效

而不是：

- 直接 panic
- 直接把 `leader member not found` 当致命错误
- 让 secondary 因 prestart 失败反复 CrashLoop

原因是这类错误很多时候并不表示：

- 当前实例数据已经不可用
- 必须立刻重建
- 或必须立刻判定整个拓扑无解

它更常见的含义只是：

- 当前这一拍还拿不到**可用的同步源**

所以在这类启动前路径里，更稳妥的一句话是：

- **没有有效 leader member 时，优先等待；只有在明确需要重建且同步源可用时，才进入 `SyncData()`。**

## 只有“继续操作确实不安全”时，才应该阻断 control-plane

控制面真正应该 fail closed 的，是下面这类情况：

- 已经有明确证据表明继续 promote 会破坏拓扑
- 已经有明确证据表明 leader 不能安全对外写
- 已经有明确证据表明关键能力处于关闭或错误状态

而不应因为下面这类情况就直接阻断：

- 某个可选插件视图里查不到行
- 某条 metadata query 受版本差异影响返回空
- 某个补强探测面暂时不可读，但主路径证据还在

## 推荐的实现口径

设计这类能力探测时，建议固定成：

1. **主判断**
- promote / has-lease 真正依赖的最小安全条件是什么

2. **主证据**
- 哪条探测路径最稳定、最能跨版本工作

3. **补强证据**
- 其他查询面只用于增强置信度，不直接决定致命失败

4. **降级策略**
- 单一探测面空结果时，是记成 unknown，还是必须 hard fail

5. **错误分级**
- “unknown”
- “disabled”
- “fatal”

如果这几层不拆开，现场就很容易变成：

- 表面看像复制或 GTID 问题
- 但最早破坏点其实是 control-plane 在能力探测上 fail closed 过早

## 推荐的实现顺序

### 1. 先收职责边界

- 列清楚谁只能报信号
- 列清楚谁有最终拍板权
- 把旧的并行决策点从代码和文档里标出来

### 2. 再收最终发布链路

- 谁确认 primary
- 谁切 endpoint / publication
- 谁做 fencing
- 谁回写完成信号

### 3. 最后处理兼容与降级

- 哪些老路径保留为兼容输入
- 哪些字段或行为需要过渡期保留
- 哪些老脚本应明确退出主路径

## 验收口径

后续验证是否真正完成 control-plane 收口时，建议固定回收以下结果：

1. primary 最终由哪个组件确认
2. endpoint 在何时切换
3. old primary fencing 是否发生，触发条件是什么
4. failover / switchover 完成信号由哪个组件给出
5. 启动期是否还会把抖动扩散成对外状态
6. 日志证据链是否能说明“谁决定了 primary、为什么在那个时点发布”
